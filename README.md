# 学生用 AI 写小游戏：过程拆解与提示词范例

面向零基础、以「Canvas 2D 跑酷类」为例（可与本仓库 Chromium 版小恐龙对照学习）。  
原则：**AI 负责生成与改错，学生负责拆需求、写提示词、运行验收、能讲清原理。**

---

## 学生自学指南（自学者从这里开始）

### 1）学前准备（一次性）

| 项目      | 说明                                                                |
| ------- | ----------------------------------------------------------------- |
| 浏览器     | 推荐 **Chrome** 或 **Edge**（方便用开发者工具）                                |
| 编辑器     | VS Code / Cursor 等均可                                              |
| 打开方式    | 建议用 **本地静态服务**打开页面（避免部分环境下 `file://` 限制）；可用 Live Server、或按课程提供的命令 |
| Git（建议） | 每完成一阶段做一次提交，便于回退；操作见 **`GitHub-Hosting-Tutorial.md`**             |

### 2）每一步固定「学习循环」（建议照做）

1. **读**：本阶段「目标」「验收」「非目标」各读一遍，用笔划掉你不打算做的功能。  
2. **写提示词**：用 §2.1 的骨架填空，优先 **需求型**，少写实现细节。  
3. **跑**：在浏览器里打开，按验收条目逐条勾。  
4. **不通则迭代**：用 §4 的「现象 + 期望」模板追问 AI，要求 **最小改动**。  
5. **记一句**：本步你改了哪个参数、画面如何变化（阶段 2 起强制养成习惯）。  
6. **提交（建议）**：`git add` / `git commit` / `git push`，说明写 `step: 阶段N …`。  
7. **下一关**：只有验收全过，再打开下一阶段。

### 3）和 AI 协作的纪律（自学最容易翻车的地方）

- **一次只做一个阶段**；提示词里写清「非目标」，避免 AI 顺手写满全套。  
- **先能玩再好看**：方块能跑通之前，不要花大量时间在美术。  
- **代码要能讲**：合入你项目前，用自己的话解释「主循环里每一帧做了什么」。  
- **保留提示词**：复制到文末附录表或你自己的笔记，便于以后复盘。

### 4）何时对照本仓库 `index.js`

- **建议时机**：你自己已经用方块版跑通 **阶段 5～6**（碰撞 + 分数）之后，再打开本仓库 `index.js` 顶部导读，对照「主循环、Horizon、Trex、碰撞、计分」。
- **不建议**：第一天就通读 2000+ 行；容易淹没在细节里。

### 5）自学完成标准（给你一条底线）

你至少应能做到：

- 说清 **主循环**（何时 `update`、何时重绘）在做什么。  
- 说清 **地面 / 玩家 / 障碍** 三类对象各管什么。  
- 能改 **一个参数**（如重力、速度上限、障碍间隔）并预测大致效果。  
- 能向同学演示 **开始 → 游玩 → 撞死 → 重开** 的完整闭环（阶段 7）。

### 6）卡住了怎么办

| 现象           | 建议                              |
| ------------ | ------------------------------- |
| 页面空白 / 控制台报错 | 从第一行红字读起；把报错全文贴给 AI，并附 `相关代码片段` |
| AI 一次改太多看不懂  | 要求「只改一个函数」或「只给 diff」            |
| 物理不对（穿地、瞬移）  | 先检查「时间是否用秒（dt）」「地面 y 是顶边还是底边」   |
| 想放弃抠官方小恐龙    | 先完成你的方块版；官方版是第二本书               |

## 1. 使用方式

1. **一次只做一小步**：每步都能在浏览器里看到变化；不要一上来「帮我写完整游戏」。
2. **每步保留提示词**：复制到本文档末尾「我的提示词记录」或作业里，便于批改与复盘。
3. **验收优先于字数**：能跑、符合本步验收标准，再进入下一步。
4. **必须亲自验收与调参**：运行结果不符合需求时，用「现象 + 期望」迭代提示词；至少改一处可调参数并记录观察。

## 2. 过程总览（建议顺序）

| 阶段  | 学生要达成的行为            | 主要游戏概念                       |
| --- | ------------------- | ---------------------------- |
| 0   | 仓库 + 本地打开页面         | 静态站点、文件结构                    |
| 1   | 画布 + 主循环            | `requestAnimationFrame`、每帧更新 |
| 2   | 地面 + 玩家方块 + 重力跳跃    | 速度、加速度、地面约束                  |
| 3   | 输入（向上方向键跳）与状态       | 事件、`preventDefault`、标志位      |
| 4   | 障碍物左移与生成            | 数组、回收、随机间隔                   |
| 5   | 碰撞与 Game Over       | AABB、游戏结束分支                  |
| 6   | 分数与加速               | 累计量、难度曲线                     |
| 7   | 状态机（菜单/游戏中/结束）      | 状态切换、重开                      |
| 8   | （可选）障碍物资源图替换         | 图片资源加载、降级兜底                    |
| 9   | （可选）方块玩家下蹲           | 输入状态、角色尺寸切换                     |
| 10  | （可选）精灵图 `drawImage` | 贴图、帧动画                       |
| 11  | （可选）音效 / 暂停         | Web Audio、Page Visibility    |

## 2.1 需求型提示词 vs 实现型提示词（写给学生）

| 类型           | 写什么                              | 尽量不写什么（交给 AI 或自己读生成结果）                       |
| ------------ | -------------------------------- | -------------------------------------------- |
| **需求型**（推荐）  | 玩家能做什么、画面要出现什么、何时算通过/失败、不能用什么技术栈 | 具体变量名、`vy += gravity * dt` 这种推导式、指定必须用哪个 API |
| **实现型**（易越界） | 「用 fillRect」「写 aabb 函数签名」        | 容易锁死思路，且模型照抄后学生仍不理解「为什么」                     |

**推荐提示词骨架**（每阶段可照抄填空）：

```text
【背景】已有：……（没有就写「从零开始」）
【目标】玩家/系统应呈现的行为：……
【验收】我打开页面后，能依次确认：①…… ②…… ③……
【约束】技术栈与范围：……（例如：仅浏览器原生、单页、不引入构建工具与框架）
【非目标】本步不要求：……（避免 AI 一次性做太多）
```

以下各阶段范例均按 **需求 + 验收 + 约束 + 非目标** 来写；实现细节由 AI 或师生在验收不通过时再迭代。

## 3. 分阶段详解 + 提示词范例（需求导向）

### 阶段 0：仓库与最小页面

**目标**：能在本机通过浏览器稳定打开一个最小站点入口；为后续画布做准备。

**验收**：打开页面有标题或说明文字；无未处理致命报错；脚本已挂载（可通过「改一行 log 能看见」自测）。

**提示词范例 A（从零）**

```text
【背景】从零开始做一个本地可打开的最小网页项目，用于后续在浏览器里做 2D 小游戏。
【目标】访问入口页时，能看到页面标题（或主标题文案）为「我的跑酷练习」；页面为后续放游戏画布做好准备（是否现在就放画布由你决定，但若放置需保证尺寸适合横条跑酷）。
【验收】① 用本地静态方式打开后控制台无红色致命错误 ② 能说明我接下来应该把游戏逻辑写在哪个文件里。
【约束】仅使用浏览器原生技术；不要引入 npm、打包器或前端框架。
【非目标】本步不要求游戏循环、不要求可玩逻辑。
```

**提示词范例 B（已有 HTML，只补「能接脚本」）**

```text
【背景】我已有或即将有一个 index.html，希望单独用 game.js 承载后续逻辑。
【目标】页面加载后能执行 game.js；页面结构尽量简单，便于以后只改 JS 迭代游戏。
【验收】① 打开页面后能在控制台看到「脚本已加载」类信息（具体文案你可自定）② 不依赖外网 CDN 才能打开（资源尽量本地）。
【约束】不引入框架与构建工具。
【非目标】本步不要求任何游戏玩法。
```

### 阶段 1：主循环 + 清屏

**目标**：让「画面会随时间持续更新」这件事成立，为后面动画打基础。

**验收**：肉眼能看出画面在持续刷新（例如整屏背景色或简单占位内容每帧更新）；可选：角落有「时间或帧」在变化，便于我确认没有卡死。

**提示词范例**

```text
【背景】已有可打开的页面，并准备用一块矩形绘制区域作为游戏画面（浏览器里常见的 2D 画布方案即可）。
【目标】实现持续运行的绘制循环：每一小段时间后画面会重新绘制一次，而不是只画一次就停住。
【验收】① 长时间开着页面，画面仍持续更新（我可录屏证明）② 若我缩小或保持窗口，不出现未捕获错误导致白屏。
【约束】不引入游戏引擎或前端框架；适合零基础单仓库维护。
【非目标】本步不要求角色、障碍、分数；不要求精确帧率数字，只要求「持续刷新」可感知。
```

---

### 阶段 2：地面 + 玩家 + 重力感

**目标**：在阶段 1「横向平移长方形」基础上，**升级为玩家角色**（不删除重写）；画面上能区分「地面」和「角色」；角色从空中落下会停在地面上，运动感觉像受重力影响（不要求按键）。

**验收**：① 能分清哪一块是地面、哪一块是角色（形状颜色随意）② 角色最终稳定站在地面上，不会无限掉出画布下方 ③ 若把初始高度抬高再运行，应先下落再停住。

**提示词范例**

```text
【背景】上一版已有持续重绘，并且当前画面里有一个“横向平移的长方形”（用于证明循环在跑）。
【目标】不要删除重写，而是把这个长方形升级为「玩家占位体」：将其主要行为从“横向平移演示”改为“受重力影响的角色”；新增可识别地面，玩家从空中下落后会在地面稳定停住。
【验收】① 玩家落地后保持在地面上 ② 我如果把起始高度改高，应看到下落过程 ③ 画面里地面与玩家边界关系合理（不要求美术，只要逻辑合理）。
【约束】仍不引入框架；时间步长与运动关系应合理，避免速度随帧率剧烈漂移；请尽量复用现有变量和结构，减少大改。
【非目标】本步不要求跳跃键、不要求障碍、不要求分数；不要求继续保留“自动横向移动演示”效果。
```

---

### 阶段 3：跳跃输入与「不能耍赖跳」

**目标**：玩家通过约定按键（向上方向键）起跳；只有在「站在地面上」时才允许起跳；避免在空中通过连续按键获得不合理加速（除非课程明确要二段跳）。

**验收**：① 地面连续多次跳跃表现正常 ② 人在空中时，按跳跃键不应再次起飞（直到再次落地）③ 若方向键触发页面滚动，应处理到不影响游玩或可接受。

**提示词范例**

```text
【背景】已有带重力与地面的玩家占位体。
【目标】玩家用键盘向上方向键起跳；仅在「确实站在地面」时响应起跳；空中按键不应产生新的起跳效果，直到落地。
【验收】① 连续玩 10 次跳跃，无「空中连跳升空」 ② 落地后立刻按向上方向键能再次跳起 ③ 若出现页面随方向键滚动，应给出可游玩方案（说明取舍即可）。
【约束】不引入框架；交互应清晰可教给零基础同学。
【非目标】本步不要求障碍、碰撞、分数；不要求下蹲或长按蓄力。
```

---

### 阶段 4：障碍从右侧来、离开屏幕要消失、会不断出现

**目标**：有「从右往左」靠近玩家的障碍；离开视野后不应永久堆积拖慢游戏；新障碍出现节奏有变化，玩起来不像完全固定节拍器（可有随机性）。

**验收**：① 连续运行一段时间，帧率感不明显恶化（或障碍数量有上限/会回收）② 至少能同时看到「旧障碍离开、新障碍出现」 ③ 障碍与地面关系合理（例如立在地上或合理高度）。

**提示词范例**

```text
【背景】已有可跳跃的玩家与左右条带状场地感（地面）。
【目标】加入「从场地右侧进入、向左运动」的障碍；障碍离开左侧视野后不应继续占用游戏；应能持续生成后续障碍，且相邻障碍之间的间隔不要完全固定（允许随机，但需在合理范围内，避免不可能通过）。
【验收】① 玩 30 秒以上仍流畅或可接受 ② 障碍不会只出现一个就停止 ③ 两次障碍之间的空隙有时大有时小，但不应频繁出现「完全贴脸无法反应」的极端（若出现请说明并给默认参数建议）。
【约束】不引入引擎；障碍外观可用简单几何占位。
【非目标】本步不要求碰撞结算、不要求 Game Over、不要求分数。
```

---

### 阶段 5：碰到就结束 + 明确反馈

**目标**：玩家与障碍发生接触时，一局结束；用户能明确感知「已结束」（画面或文字或停止运动等任选组合）。

**验收**：① 故意撞一次必结束 ② 结束后世界不应继续像进行中那样推进（或应有清晰冻结态）③ 我能向同学口头说明「怎样算撞到了」。

**提示词范例**

```text
【背景】已有移动障碍与可操控玩家。
【目标】定义「失败」：玩家与障碍发生接触时本局结束；结束后应给出清晰反馈（例如中央提示、变暗、停止更新等，任选但需一致）。
【验收】① 我主动撞障碍必触发结束 ② 结束后障碍与玩家不应继续按「进行中」逻辑无限跑下去 ③ 反馈文案或状态可被零基础同学看懂。
【约束】碰撞判定用你认可的 2D 常见近似即可，但需在注释里用一句话说明「判定依据是什么形状」。
【非目标】本步不要求重新开始一局、不要求记录历史最高分、不要求复杂粒子特效。
```

---

### 阶段 6：分数与「越玩略难一点」

**目标**：游玩过程中有「进展感」的量化显示（分数或距离）；随时间或分数推进，整体压力略增，但要有上限或缓和机制，避免瞬间不可玩。

**验收**：① 分数（或等价进度条）在游玩中持续增长或更新 ② 能感知难度变难，但不会在前几秒就 impossible ③ 结束后仍能看到本局结果（可与下一阶段合并，若本步未做菜单也可先显示在结束画面上）。

**提示词范例**

```text
【背景】已有可玩的一局：移动障碍、跳跃、碰撞失败与结束反馈。
【目标】增加「本局进度」的可见量化（分数或跑过的距离，二选一或同时都可，但需解释含义）；随着游玩进行，整体难度应缓慢上升，并设有上限或封顶策略，避免无限加速导致不可玩。
【验收】① 我站着不动也会因障碍到来而失败，但前几秒应给我学习反应时间（你可调参）② 分数或进度在游玩中可见变化 ③ 难度上升我能用一句话向同学解释「依据什么在变难」。
【约束】不引入后端；数据仅存前端内存即可。
【非目标】本步不要求排行榜云存储；不要求多关卡地图切换。
```

---

### 阶段 7：开始、结束、再来一局（无需刷新网页）

**目标**：有明确的「还没开始 / 进行中 / 已结束」；**开始游戏按空格键触发**；结束后满足重开条件并可用约定按键开新一局；全程尽量不必 F5 刷新。

**验收**：① 第一次打开时有明确提示，按空格键可开始 ② 进入结束状态后按 **R** 键可重开 ③ 再来一局时障碍与分数应重置到一致的起始规则。

**提示词范例**

```text
【背景】已有单局玩法、结束反馈、分数与障碍系统。
【目标】把体验整理成可重复游玩的流程：未开始时提示“按空格键开始”；进行中专注操作；结束后提示“按 R 键重开”。重开条件为：仅在结束状态（GAMEOVER）下，按 R 键才触发新一局；重开时应重置本局相关状态（障碍、分数、玩家姿态等），且不需要我手动刷新浏览器页面。
【验收】① 在未开始状态按空格键可进入游戏 ② 在结束状态按 R 键可重开，且在进行中误按 R 不应重置当前局 ③ 连续玩 3 局，每局规则一致 ④ 每局开始前的提示与结束后的提示不互相矛盾 ⑤ 若我误按键，不应把游戏锁进无法恢复的状态。
【约束】仍保持单页原生实现；状态切换逻辑应可读（欢迎用清晰命名或简短状态说明）。
【非目标】本步不要求存档到第二天、不要求账号系统。
```

### 阶段 8（可选）：把障碍物从「色块」换成图片

**目标**：将障碍物渲染从几何方块替换为独立 PNG，同时保持已有玩法逻辑不变（生成、速度、碰撞、状态机流程不变）。

**验收**：① 游戏正常启动无报错 ② 仙人掌和翼龙都能显示为图片 ③ 翼龙有两帧切换动画 ④ 碰撞结果与原来一致可用 ⑤ 开始/结束/重开流程不受影响。

**提示词范例**


## 阶段11补充：音效实现原理与最小落地模板

这一节是给完成阶段11的同学做“原理对齐”，帮助你知道为什么声音能响、什么时候会不响。

### A. 这套项目里的音效实现思路（对照 `index.html` + `index.js`）

1. **资源放置**：在 `index.html` 的 `<template id="audio-resources">` 中放 3 个 `<audio>`（按键、撞击、得分），音频是 `data:audio/mpeg;base64,...` 内嵌资源。  
2. **首次交互后初始化**：在玩家首次按键开始时，创建 `AudioContext` 并解码音频（浏览器自动播放策略要求用户手势）。  
3. **缓存音频缓冲**：把解码后的 `AudioBuffer` 存在对象里（例如 `soundFx.xxx`）。  
4. **事件触发播放**：起跳、撞击、里程碑时，创建 `BufferSource` 播放对应缓冲。  

### B. 最小落地模板（教学版）

> 目标：先做到“能稳定播放起跳音效”，再扩展撞击和得分音效。

```text
【背景】当前游戏已有开始/进行中/结束状态与基本输入逻辑，但还没有稳定音效。
【目标】基于浏览器原生能力，实现最小音效链路：用户首次交互后初始化音频；起跳时能播放短音；后续可扩展撞击与得分音效。
【验收】① 首次按键后音频可用 ② 起跳时能稳定听到反馈 ③ 未完成初始化时不会报错中断游戏 ④ 不破坏现有开始/结束/重开流程。
【约束】仅原生JS；最小改动；优先复用现有事件触发点；对“浏览器自动播放限制”给出一句说明。
【非目标】本步不要求背景音乐、不要求音量设置面板、不要求复杂混音。
【输出要求】先给改动点清单，再给关键函数（初始化、播放、触发点）替换代码，并附4条手动验收步骤。
```

### C. 常见问题速查

| 现象 | 原因 | 处理建议 |
|------|------|----------|
| 按键后没声音 | 未在用户手势后初始化音频 | 把初始化放在首次 `keydown/click/touchstart` 里 |
| 偶尔有声偶尔无声 | 播放时缓冲尚未准备好 | 播放前判空，未就绪时安全返回 |
| 连续触发有卡顿 | 每次处理方式不当 | 使用标准 `createBufferSource -> connect -> start` 流程 |
| 切换标签页后行为异常 | 暂停/恢复规则不一致 | 明确 hidden 时冻结，visible 时继续或提示继续（二选一） |
